# First Principles Thinking - 제1원리 기반 문제 분석

대상: `$ARGUMENTS`

---

## 프레임워크 개요

**First Principles Thinking**(제1원리 사고)은 문제를 가장 근본적인 진실(first principles)로 분해한 뒤, 그로부터 해결책을 처음부터 재구성하는 문제 해결 모델입니다.

이는 기존의 관행, 가정, 유추(Reasoning by Analogy)에 의존하지 않고, "이것이 정말 사실인가?"를 끊임없이 질문합니다.

> "I think it's important to reason from first principles rather than by analogy. The normal way we conduct our lives is we reason by analogy. [But] with first principles, you boil things down to the most fundamental truths... and then you reason up from there." — *Elon Musk*

### 역사적 맥락

- **아리스토텔레스**: "사물이 알려지는 최초의 근거"로 제1원리를 정의 (*형이상학*)
- **물리학**: *ab initio* 계산 — 경험적 가정 없이 확립된 물리 법칙에서 직접 출발

### 핵심 비유: 셰프 vs 요리사

- **요리사 (유추 사고)**: 레시피를 따름. 레시피가 없으면 요리 불가. *무엇이* 작동하는지는 알지만 *왜*인지는 모름.
- **셰프 (제1원리 사고)**: 맛의 원리와 화학을 이해함. 어떤 재료든 새로운 레시피를 만들 수 있음.

### 참조 사례: SpaceX

- **문제**: 로켓 구매 비용 $65M+
- **유추 사고**: "우주 탐사는 정부만 할 수 있을 만큼 비싸다"
- **제1원리**: 로켓 = 항공우주 알루미늄 + 티타늄 + 구리 + 탄소섬유. 원자재 가치 ≈ 로켓 가격의 ~2%
- **결론**: 비용은 공급망과 조립에 있다 → 자체 제작 + 재사용

## Step 1: 가정(Assumptions) 식별 및 도전

대상 문제/주제에 대해 정보를 수집하고, 현재 존재하는 **가정**들을 모두 나열합니다.

### 정보 수집

- 코드베이스 관련 문제 → Grep, Glob, Read로 현황 파악
- 기술적 문제 → WebSearch로 현재 업계의 일반적 접근법 조사
- 비즈니스/프로세스 문제 → 현 상태의 수치와 제약 조건 파악

### 가정 추출 기법: 소크라테스식 질문

각 가정에 대해 아래 질문을 적용:

1. **명확화**: "이것이 정확히 무엇을 의미하는가?"
2. **근거**: "왜 이것이 사실이라고 믿는가? 어떤 증거가 있는가?"
3. **대안**: "다르게 생각할 수는 없는가? 반례는?"
4. **결과**: "이 가정이 틀리다면 어떤 결과가 생기는가?"
5. **원초적 질문**: "애초에 왜 이것이 필요한가?"

### 출력 형식

```markdown
## 🔍 가정 목록

| # | 가정 | 출처/근거 | 도전 질문 |
|---|------|----------|----------|
| 1 | [현재 당연하게 여기는 것] | [이 가정의 출처] | [이것이 정말 사실인가?] |
| 2 | ... | ... | ... |
```

---

## Step 2: 근본 진실(Fundamental Truths)로 해체

Step 1에서 식별한 가정들을 걷어내고, **더 이상 분해할 수 없는 근본적 사실**만 남깁니다.

### 해체 기준

- **물리적 제약**: 변경 불가능한 자연법칙, 하드웨어 한계
- **논리적 필연**: 수학적/논리적으로 반드시 참인 것
- **측정 가능한 사실**: 실제 데이터로 검증된 수치
- **핵심 요구사항**: 이것 없이는 문제 자체가 성립하지 않는 것

### 해체 기법: 5 Whys

```
가정: "배포에 30분이 걸린다"
  → Why? CI 파이프라인이 느리다
    → Why? 테스트가 전부 순차 실행된다
      → Why? 병렬화가 설정되지 않았다
        → Why? 테스트 간 상태 의존성이 있다
          → Why? 테스트가 공유 DB를 사용한다
            → 근본 진실: "테스트가 공유 상태에 의존한다"
```

### 출력 형식

```markdown
## 🧱 근본 진실

| # | 근본 진실 | 유형 | 검증 방법 |
|---|----------|------|----------|
| 1 | [더 이상 분해 불가능한 사실] | 물리적/논리적/측정된/필수 | [어떻게 확인했는가] |
| 2 | ... | ... | ... |
```

## Step 3: 새로운 해결책 재구성

Step 2의 근본 진실만을 재료로, **기존 방식을 무시하고** 처음부터 해결책을 설계합니다.

### 재구성 원칙

1. **기존 방식 참조 금지**: "원래 이렇게 했으니까"는 근거가 아님
2. **근본 진실만 사용**: Step 2에서 검증된 사실만 빌딩 블록으로 사용
3. **제약 조건 최소화**: "할 수 없다"를 의심하되, 물리적 제약은 존중
4. **10x 목표**: 10% 개선이 아닌 10배 개선을 목표로 사고

### 출력 형식

```markdown
## 💡 재구성된 해결책

### 핵심 아이디어
[한 문장으로 요약]

### 근거 매핑
| 근본 진실 | → | 해결책 요소 |
|----------|---|-----------|
| [Step 2의 진실 #N] | → | [이 진실에서 도출된 설계] |

### 구체적 실행 방안
1. [첫 번째 단계]: [상세 설명]
2. [두 번째 단계]: [상세 설명]
3. ...

### ⚖️ 트레이드오프
| 얻는 것 | 잃는 것 | 수용 가능한 이유 |
|---------|---------|----------------|
| ... | ... | ... |
```

## Step 4: 종합 정리

전체 분석을 하나의 흐름으로 요약합니다.

```markdown
## 📋 제1원리 분석 요약

### 문제
[원래 문제 1줄 요약]

### 깨진 가정
[가장 영향력 있었던 가정 1~2개]

### 발견된 근본 진실
[핵심 진실 1~2개]

### 제안 해결책
[최종 해결책 요약]

### 기존 접근 대비 차이점
[유추 사고 방식과의 핵심 차별점]
```

## 적용 판단 기준

이 스킬은 **정신적으로 비용이 높은** 사고법입니다. 모든 문제에 적용하지 마세요.

**적합한 경우:**
- 10% 개선이 아닌 **10x 혁신**이 필요할 때
- 현재 "모범 사례"가 **실패**하고 있을 때
- 아무도 해본 적 없는 **새로운 영역**일 때

**부적합한 경우:**
- 사소한 결정 (일상적 선택)
- 이미 잘 작동하는 시스템의 미세 조정
- 시간 제약이 극도로 심한 상황
